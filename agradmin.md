# Админ-панель агрегатора (agradmin)

Админ-панель (`agradmin`) -- веб-интерфейс для управления платёжным агрегатором. Построена на Yii2 и взаимодействует с ядром агрегатора (`aggregator`, Laravel) через REST API.

> Подробные описания полей доступны прямо в интерфейсе -- наведите курсор на иконку **?** рядом с названием поля или колонки.

---

## 1. Администраторы

Раздел для управления учётными записями пользователей админ-панели.

Каждому администратору задаются логин, email и пароль. Поддерживается двухфакторная аутентификация (MFA) через TOTP-приложение (Google Authenticator и аналоги). При включении MFA пользователю показывается QR-код для привязки, после чего при каждом входе запрашивается одноразовый код.

### Назначение ролей

Роли управляют доступом к разделам и действиям (RBAC). На странице назначений для каждого администратора отображаются доступные и назначенные роли -- можно перемещать роли между списками.

---

## 2. Мерчанты

Мерчанты подключённые к платёжной системе. Это центральная сущность агрегатора: к мерчантам привязываются методы оплаты, балансы, комиссии и настройки уведомлений.

### Создание мерчанта

При создании задается имя и генерируются идентификационные данные. При необходимости добавляются URL-ы для взаимодействия (callback, success, fail, flow, logo), настройки выплат и Telegram-уведомлений. Здесь же выбираются вертикали (направления бизнеса) и IP-адреса для whitelist.

### Редактирование мерчанта

Страница редактирования содержит несколько блоков:

- **Основные настройки** -- все поля из создания, плюс статус, контактные данные (email, телефон, ФИО), общие комиссии, агентские комиссии, часовой пояс, таймаут запросов, настройки обратной интеграции (reverse gateway).
- **Пополнение баланса** -- ручное пополнение баланса мерчанта с указанием суммы и хеша транзакции.
- **Настройки типов методов (MethodTypeRate)** -- для каждого типа метода (c2c, sbp и т.д.) задаются индивидуальные комиссии payin/payout, режим каскада (по шансам или по порядку), источник курса и другие параметры.
- **Методы** -- таблица привязанных методов оплаты с индивидуальными комиссиями, лимитами сумм, шансами попадания в каскад и источниками курса. Методы можно включать/отключать, редактировать ставки, менять порядок в каскаде.

Все изменения записываются в историю аудита. При сохранении можно указать причину изменения.

---

## 3. Методы оплаты

Метод оплаты -- конкретный канал приёма/выплаты денег, привязанный к провайдеру (gateway). Например: "OnePAY_СБП_<1000" -- метод через провайдер OnePAY для СБП-платежей до 1000.

### Создание и редактирование

Метод характеризуется кодом, валютой, внутренним именем и привязкой к провайдеру. При выборе провайдера динамически подгружаются его параметры (API-ключи, секреты и т.д.).

Ключевые настройки:
- **Комиссии** payin/payout -- процент провайдера.
- **Лимиты** payin/payout -- максимальная сумма оборота, при достижении которой трафик перестаёт идти на метод.
- **Время жизни заявки** -- через сколько минут заявка получает статус "Истёк срок".
- **Шаг разброса** -- единица изменения суммы для маскировки (1, 10, 100...). Работает в связке с "суммой разброса" мерчанта.
- **Работа по контрагентам** -- если клиент уже платил через этот метод, направлять его сюда в первую очередь.
- **Защита от минусовых заявок** -- блокировка заявок, которые приведут к убытку по курсу.
- **Чек обязателен** -- плательщик должен загрузить подтверждение оплаты.

### Привязка мерчантов

В нижней части страницы метода -- таблица привязанных мерчантов с индивидуальными ставками. Можно массово менять лимиты сумм и привязывать новых мерчантов через модальное окно.

---

## 4. Типы методов

Типы методов -- это категории, объединяющие методы оплаты по способу (c2c, sbp, crypto и т.д.). У каждого типа есть техническое название, отображаемый алиас и логотип.

Флаг **"Требует доп форму"** определяет поведение при создании платежа: если включен, инвойс не создаётся сразу -- вместо этого генерируется ссылка на форму, где плательщик вводит дополнительные данные.

---

## 5. Провайдеры (шлюзы)

Провайдер (gateway) -- подключение к внешней платёжной системе. Каждый провайдер имеет namespace (определяет класс-обработчик) и набор параметров, уникальных для каждой платёжной системы (API-ключи, URL-ы, секреты).

### Настройки провайдера

- **Таймауты** -- максимальное время ожидания ответа от провайдера для payin и payout.
- **Telegram-интеграция** -- три чата: для диспутов, для уведомлений о статусе "На проверке" и для ProviderBot (бизнес-чат с провайдером).
- **Агрегация лимитов** -- если включена, лимиты считаются суммарно по всем методам провайдера.
- **Whitelist IP** -- список IP-адресов, с которых провайдер отправляет вебхуки.

### Лимиты

На вкладке лимитов отображаются текущие лимиты по валютам с возможностью массового изменения. Лимиты считаются от даты последнего сброса.

---

## 6. Группы провайдеров

Группы объединяют несколько провайдеров для агрегации лимитов и балансов. Режим баланса определяет, как рассчитывается общий показатель группы: суммирование балансов всех провайдеров или единый баланс.

Для каждой группы задаются лимиты по валютам (payin/payout). Провайдер может состоять только в одной группе.

---

## 7. Офферы методов

Раздел для учёта коммерческих предложений от платёжных провайдеров. Позволяет систематизировать информацию о доступных предложениях: условия по комиссиям (payin/payout), лимиты транзакций, валюты расчётов, тип интеграции, принимаемые вертикали и т.д.

Офферы можно фильтровать, экспортировать и привязывать к конкретным методам. За каждым оффером закреплён менеджер.

---

## 8. Транзакции

### Список транзакций

Основная рабочая таблица с платежами и выплатами. Поддерживает фильтрацию по всем ключевым параметрам: ID, даты, тип операции (payin/payout/withdrawal), метод, провайдер, мерчант, статус, причина, реквизит.

Доступен экспорт в CSV с выбором разделителя и возможностью добавить строки балансов. Опция "Только с инвойсом" фильтрует платежи, у которых есть привязанный инвойс.

### Редактирование транзакции

Карточка транзакции содержит информационные (только для чтения) и редактируемые поля. Можно менять сумму, курс, комиссию, статус и причину. Доступны действия:
- Загрузка чеков (скриншотов оплаты)
- Отправка сообщений в чат поддержки
- Передача выплаты на другой метод
- Повторная отправка callback мерчанту
- Проверка статуса у провайдера

### Споры

Отдельная страница со списком транзакций в статусе "спор". Те же фильтры, что и в основном списке, плюс фильтр по наличию чека и по дате обновления.

## 10. Контрагенты

Контрагент -- конечный плательщик, идентифицированный по совокупности данных (fingerprint, телефон, email, IP, userId).

При включенной опции "Работа по контрагентам" на методе, система запоминает привязку плательщика к конкретному методу и в будущем направляет его на тот же метод.

На странице контрагента отображается история платежей с суммами и статистикой. Доступные действия: блокировка/разблокировка, отвязка от метода, перепривязка к другому методу.

---

## 11. Заблокированные пользователи

Список клиентов, заблокированных на уровне конкретного мерчанта. Блокировка запрещает пользователю совершать платежи через данного мерчанта. Можно фильтровать по ID, пользователю и мерчанту. Единственное действие -- разблокировка.

---

## 12. Настройки приложения

Централизованные настройки агрегатора, сгруппированные по категориям. Значения хранятся в таблице `settings` и кэшируются.

### Платежи

| Ключ | Тип | Описание |
|------|-----|----------|
| `app.time_limit` | int | Время жизни платежа (сек). После истечения -- статус expired |
| `app.dispute_expired_days` | int | Срок подачи апелляции (дни). Сколько дней можно открыть спор |
| `app.callback_expired_hours` | int | Приём колбеков (часы). Старые колбеки игнорируются |
| `app.antispam_time_interval` | int | Антиспам: временное окно (мин) для подсчёта платежей от одного пользователя |
| `app.antispam_payments_limit` | int | Антиспам: макс. платежей в окне. При превышении -- блокировка |
| `app.payment_failed` | bool | Сохранять неудачные платежи. ON = FAILED для аналитики, OFF = удаляем |
| `app.auto_check_payment` | bool | Автопроверка: опрашивать провайдеров для обновления статусов |

### Хранение данных

| Ключ | Тип | Описание |
|------|-----|----------|
| `kafka.retention.failed_payments_days` | int | Хранение failed платежей в PostgreSQL (дни). Затем -- в Kafka. Рекомендуется: 7 |
| `kafka.retention.cascade_attempts_days` | int | Хранение cascade_attempts в PostgreSQL (дни). Рекомендуется: 7 |

### Аналитика

| Ключ | Тип | Описание |
|------|-----|----------|
| `clickhouse.failed_payments_ttl_days` | int | TTL данных в ClickHouse (дни). Рекомендуется: 90 |

### Telegram

| Ключ | Тип | Описание |
|------|-----|----------|
| `app.reportbot_token` | string | Токен бота отчётов (лимиты, автосписания, выводы). Кнопка "Привязать webhook" |

| `app.telegram_logger` | string | Токен бота логгера (технические логи и ошибки) |
| `app.telegram_logger_id` | string | Chat ID для технических логов |
| `app.telegram_limit_chat_id` | string | Chat ID для уведомлений о лимитах провайдеров |
| `app.telegram_admin_chat_id` | string | Chat ID для уведомлений о выводах и админ-событиях |
| `app.telegram_fin_chat_id` | string | Chat ID для уведомлений о просроченных выплатах |

### Support Service

| Ключ | Тип | Описание |
|------|-----|----------|
| `app.sup_service` | bool | Включить интеграцию с сервисом поддержки и трейдер-ботом |

| `app.support_service_token` | password | Токен аутентификации в сервисе поддержки |

### Прочее

| Ключ | Тип | Описание |

## 13. Курсы валют

Отображение текущих обменных курсов с различных бирж. Таблица содержит валютную пару, источник (биржу), тип операции (покупка/продажа) и текущий курс. Раздел только для просмотра.

---

## 14. Экспорт

Список сгенерированных CSV-файлов (из разделов транзакций). Файлы в статусе "В обработке" или "Загружено". Готовые файлы можно скачать -- при загрузке отображается прогресс-бар с возможностью отмены.

---

## 15. История изменений

Аудит-лог всех изменений сущностей (мерчантов, методов, ставок). Для каждой записи отображается: действие, дата, пользователь, изменённые поля (старое и новое значение) и указанная причина изменения.

Доступ к истории -- через кнопку в карточке редактирования сущности.

---

## 16. Логи

Системные логи действий администраторов: ID записи, ID администратора и выполненное действие.
