# Статусы транзакций

> Актуально для: Aggregator (паyin / payout). Источник: `app/Services/OrderStatus.php`, `SubStatusSeeder.php`, `CallbackController.php`.

---

## Payin (входящий платёж)

| Статус | Значение в API | Отображение | Описание |
|--------|---------------|-------------|----------|
| `created` | `created` | Создано | Платёж только что создан, трейдер ещё не принял |
| `pending` | `pending` | В процессе | Трейдер принял, ждём подтверждения перевода |
| `in_check` | `in_check` | На проверке | Трейдер отметил платёж — идёт ручная проверка |
| `dispute` | `dispute` | Апелляция | Открыт диспут (мерчантом или клиентом) |
| `finished` | `finished` | Оплачено ✅ | Платёж успешно завершён |
| `canceled` | `canceled` | Отменено ❌ | Платёж отменён (см. sub-status ниже) |
| `expired` | `expired` | Истёк срок | Время ожидания вышло, платёж не завершён |
| `failed` | `failed` | Неудачно | Техническая ошибка на стороне провайдера |

### Жизненный цикл платежа

```
created → pending → in_check → finished
                             ↘ canceled
                   ↘ dispute → finished
                             ↘ canceled
         ↘ expired
         ↘ failed
```

**Активные статусы** (платёж ещё в работе): `created`, `pending`, `dispute`, `in_check`

## Sub-статусы

Sub-статус даёт дополнительный контекст к основному статусу.

### При `canceled`

| reason | Что произошло |
|--------|--------------|
| `fake check` | Трейдер получил поддельный чек от клиента |
| `no transfer made` | Клиент не совершил перевод |
| `incorrect details` | Неверные реквизиты (банк, сумма, счёт) |

### При `dispute`

| reason | Что произошло |
|--------|--------------|
| `dispute created by merchant` | Мерчант открыл диспут |
| `dispute created by client` | Клиент открыл диспут |
| `assigned to executor` | Назначен исполнитель |
| `support assigned after 30 minutes` | Саппорт назначен автоматически через 30 мин |
| `manager assigned after 60 minutes` | Менеджер назначен автоматически через 60 мин |
| `request for video/statement from merchant` | Запрошено подтверждение от мерчанта |
| `request for video/statement from executor` | Запрошено подтверждение от трейдера |

### При `finished`

| reason | Что произошло |
|--------|--------------|
| `success` | Штатное завершение |
| `success with recalculation` | Завершён с перерасчётом суммы |
| `forced success` | Принудительно завершён (операция поддержки) |

## Payout (выплата)

Выплаты используют то же поле `state`, но логика немного отличается.

| Статус | Описание |
|--------|----------|
| `pending` | Выплата создана, ожидает обработки |
| `in_check` | На проверке у трейдера |
| `finished` | Выплата успешно отправлена ✅ |
| `canceled` | Выплата отменена |
| `expired` | Истёк срок обработки |

> ⚠️ **Важно:** Коллбэк на отмену выплаты (`canceled`) отправляется **только если** у мерчанта включена настройка `send_cancel_payout_callback`. По умолчанию — выключена.

## Коллбэки — статусы в запросе

При отправке коллбэка на `callback_uri` мерчанта поле `status` содержит:

| Значение | Когда |
|----------|-------|
| `Success` | Платёж завершён (`finished`) |
| `Decline` | Платёж отменён (`canceled`) |
| `In Check` | Платёж на проверке (`in_check`) |
| `Expired` | Истёк срок (`expired`) |

> 💡 Обратите внимание: значения в коллбэке (`Success`, `Decline`) отличаются от внутренних API-статусов (`finished`, `canceled`).

## FAQ — частые вопросы

**Q: Выплата в статусе `canceled`, но деньги до клиента дошли — почему?**
Статус `canceled` означает что **система** отменила выплату (например, таймаут или ошибка). Фактический перевод мог уйти через провайдера до отмены. Нужно проверить в Grafana по `payment_id` был ли реальный перевод у провайдера.

**Q: Платёж завис в `pending` / `in_check` — что делать?**
1. Проверить, назначен ли трейдер
2. Если трейдер не отвечает > 30 мин — диспут назначается автоматически
3. Для ручного вмешательства — создать тикет в Jira (проект PAY, тип: Support)

**Q: Почему у мерчанта статус `canceled`, а в нашей системе `finished`?**
Возможны три причины:
- Мерчант смотрит на старый коллбэк (не последний)
- Коллбэк не дошёл (проверить логи по `callback_uri`)
- Статус обновился после отправки коллбэка — нужен повторный запрос статуса через API

**Q: Что значит `dispute` — деньги возвращаются?**
Нет. `dispute` — это внутренний статус проверки, деньги при этом не возвращаются автоматически. Исход диспута определяет менеджер: закрыть как `finished` или `canceled`.
